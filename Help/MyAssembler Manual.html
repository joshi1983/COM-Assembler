Syntax Rules

  • Comments
    • can be made using the ; symbol or the // symbols.
    • multilined comments can be done starting with a "comment *" line. The last line of the comment can only contain a single "*"
  • Data can be defined using db and dw.
  • Labels
    • can be defined using the : after the name of the label.
    • You can also just use a space before a data statement.
    • Refer to the offset of a label using the word offset before the label. For example, move the address of label "x" into ax using this code... "mov ax, offset x"
  • Memory References a few addressing modes are supported. Click here for more information.
  • Opcodes use Intel syntax. First is the opcode name, if any operands are required, followed by the first operand and the second operand. The specific opcodes and operands are determined in instruction set files. You can view an alphabetical listing of the instruction set by clicking View -> Opcode Listing
  • Immediate values can be in hex, binary, decimal, octal, and character format.
    • Negative signs can be placed before the numbers to make them negative. This works for hex, octal, decimal, and binary.
    • Hex, is indicated by placing the letter 'h' after your number. Hex can also be represented in the c++ style starting with '0x' followed by the hex digits.
    • Octal is represented by using the letter 'o' at the end of the digits.
    • Binary is represented by the letter 'b' being placed at the end of the digits.
    • Decimal are represented the usual way. You can also use a trailing letter "d". 10 in decimal is the default radix, number base, so you don't need anything after the number's digits unless you set the radix to something else.
    • For characters, use a ' before and after the character with the character between.
    • For strings, use a ' before and after the character string.
  • For a grammar of a line of assembly code, click here

New and future changes

New Changes
  • support for memory references
  • disassembling memory references
  • disassembling is working for some programs that don't use data definitions
  • eliminated a bug that happens when you use spaces in a memory reference
  • eliminated a bug that happens when you try using ',' as an immediate value
  • assembles more memory references ie. "WORD PTR [SI+0100h]" This way, after disassembling a program that uses them, it can be reassembled without problems.
  • an error was fixed when exporting to html and using < and > signs in the code.
  • eliminated a bug that eliminates multiple spaces from character strings
  • eliminated a bug with memory references that use several constant terms
  • eliminated a bug with byte numbers for the assemble process document counting up to 0 instead of from 0 to the end of the program
  • eliminated a bug that happens when you use a ',' in a character string
  • support for command line options
  • support for labeling data without using a :
  • starting to add a directive for switching case sensativity on but not working yet
  • a little error check to prevent loading instruction set files more than once using the include directive
  • error assembling the statement "push 5". It assembles the same as "push gs"
  • eliminated a bug that happens when you use a ':' when you aren't defining a label for example " mov al,':' "
  • eliminated a bug that happens when a numarical operand is not an immediate value. ie. shr reg16,"1" in the instruction set files. shr ax,1 will assemble right but shr ax,2 will assemble exactly the same way.
  • support for multilined comments
  • a check for duplicate instructions loaded from the instruction set files
  • instruction set files basically complete except for instructions using 3 operands
  • stopped the assembler from loading instruction set files twice
  • support for a radix directive used in Tasm example line ".Radix 10h" This sets the default number base. The base used when there is no letter left behind.
  • all addressing modes are supported however there are no multiple passes for optimization
  • eliminated a bug that happens when outputting to html lines of code with : where there is no label
  • support for segment overrides (ie. a memory reference statement like... "byte ptr[es: si]")
  • support for instructions that use 3 operands
  • using multiple passes to find the optimal memory addressing mode
Future Changes
  • bug with some registers being confused with immediate values.. ie. cmp al,dh (cmp reg8,reg8)-> cmp al, 0Dh (cmp reg8,i8)
  • bug after disassembling a file and closing the program, raises InvalidPointer or EAccessViolation
  • "sub di, 2" doesn't seem to do anything. This problem will be checked out.
  • support for using " quote signs to define character strings instead of just '
  • support for DUP in data definitions that duplicates data
  • should show an error message when a label is used that isn't defined
  • support for memory references that don't use square brackets but instead just use the name of a label
  • support for putting the prefix and assembly statement on the same line
  • adding a way of defining the type of "opcode" in the instruction sets. This can be useful in defining prefixes differently than normal opcodes so they can be assembled properly. There may be types of opcodes like "jump", "call"... That can help with jump detection and make it so data sections can be jumped over in the disassembling process. This would prevent the need for the rest of the program to be put into a hex dump instead of being properly disassembled. Labels could be created and refered to instead of constant numbers.
  • support for the EQU and = directives used in most assemblers. Also, possibly #define as used in c/c++ and rc files
  • more instructions being added to the instruction set files
  • support for procedures
  • support for structures
